# Application Load Balancer Overview

Application Load Balancer Layer 7 (Application Layer) load balancer hai, ye Layer-7 par kaam karta hai aur HTTP/HTTPS requests ko receive karke unhe intelligent routing rules ke basis par multiple targets, jaise EC2 Instances, Containers ya IP Addresses, mein distribute karta hai.

<br>
<br>

### Sabse Pehle Load Balancer Kya Hota Hai?

**Application Load Balancer** samajhne se pehle ye samajhna zaroori hai ki **Load Balancer** hota kya hai.

Maan lo tumhare paas sirf ek EC2 Instance hai aur uske upar ek web application chal rahi hai.
```
Users

↓

EC2 Instance

↓

Application
```

Agar 10 users request bhejte hain to ek EC2 instance un requests ko handle kar lega.

Lekin agar ek saath 10,000 users request bhejne lagen to?

Ek hi EC2 instance ke resources (CPU, Memory, Network) limited hote hain.

Agar uski capacity se zyada requests aa gayi to:
- Response slow ho jayega.
- CPU 100% ho sakta hai.
- Memory full ho sakti hai.
- Application crash ho sakti hai.
- Users ko timeout ya 5xx errors mil sakte hain.

Isliye production applications mein sirf ek server use nahi kiya jata. Balki multiple servers ko use kiya jata hai.

<br>

**Multiple Servers Ka Concept**:

Is problem ko solve karne ke liye application ko multiple EC2 instances par deploy kiya jata hai.
```
EC2-1

EC2-2

EC2-3

EC2-4
```

Ab application 4 servers par run kar rahi hai.

Lekin ab ek nayi problem aa gayi.

User ko kaise pata chalega ki request EC2-1 par bhejni hai ya EC2-3 par?

Kya user manually server choose karega?

Bilkul nahi.

Isi problem ko solve karta hai **Load Balancer**.

<br>

**Load Balancer Ka Main Kaam**

Load Balancer users aur backend servers ke beech mein baithta hai.
```
                   Users

                     │

                     ▼

               Load Balancer

          /          |          \

         ▼           ▼           ▼

      EC2-1       EC2-2       EC2-3
```

Ab users sirf Load Balancer ko request bhejte hain. Load Balancer decide karta hai ki request kis backend server ko forward karni hai.

Isliye users ko backend infrastructure ke baare mein kuch pata hone ki zarurat nahi hoti.

<br>
<br>

### Application Load Balancer (ALB) Kya Hota Hai?

AWS mein different types ke Load Balancers hote hain unmein se ek hai: **Application Load Balancer (ALB)**.

Application Load Balancer OSI Model ke Layer 7 (Application Layer) par kaam karta hai. Aur traffic ko backends par distribute karta hai.

Ye Layer 7 protocols ko samajhta hai.

Jaise:
- HTTP
- HTTPS
- HTTP/2
- WebSocket

Iska matlab ALB request ke andar ka application-level data dekh sakta hai.

Jaise:
- URL Path
- Host Header
- HTTP Method
- Query String
- HTTP Headers

Isi wajah se ALB ko Application-aware Load Balancer kaha jata hai.

Traditional Network Load Balancers (Layer 4) sirf IP addresses aur TCP/UDP port numbers ko dekh kar routing decision lete hain. Unhe yeh nahi pata hota ki incoming packet ke andar kya message chipa hai.

Lekin ALB Layer-7 par kaam karta hai, Layer 7 par kaam karne ka matlab hai ki ALB HTTP request ko parse karta hai. Matlab HTTP request ke ander ke data ko dekhta hai. Jaise, URL Path, Host Header, HTTP Method, Query String, HTTP Headers jaise data ko analyze karta hai aur is data ke basis par request ko healthy backends par distribute karta hai.

<br>
<br>

### ALB Ka Main Purpose

ALB ka main purpose sirf load distribute karna nahi hai.

ALB ka actual purpose hai:
- HTTP/HTTPS traffic receive karna.
- Requests inspect karna.
- Rules evaluate karna.
- Correct Target Group choose karna.
- Healthy Target ko request forward karna.

Yaani ALB ek intelligent traffic router hai.

<br>
<br>

###  Step-by-Step Working Mechanism (Traffic Flow)

AWS Application Load Balancer (ALB) ka traffic flow aur working mechanism step-by-step.

Traffic flow ki shuruaat user ke browser se hoti hai. Jab koi user aapki application ko access karne ki koshish karta hai, toh backend mein ye steps hote hain:

<br>

**Stage 1: Client-Side Request & DNS Resolution**:
- Sabse pehle, user apne browser mein website ka URL type karta hai, jaise ```https://example.com```.
- Website tak pauchne ke liye browser ko uska IP Address chiaye hota hai. Browser IP address ke liye DNS Lookup karta hai.
- Browser sabse pehle apne local cache mein check karta hai ki kya ```https://example.com``` ka IP address pehle se maujood hai. Agar nahi milta, toh wo OS (Operating System) ke resolver ko request bhejta hai.
- OS request ko DNS Server (jaise AWS Route 53) ke paas bhejta hai.
- AWS mein ALB ka apna ek fixed IP nahi hota, balki ek unique DNS name hota hai (jaise ```://amazonaws.com```). Aapka domain (```://example.com```) DNS mein ek **CNAME** ya **Route 53 Alias record** ke zariye is ALB ke DNS name par pointed hota hai.
- ALB hamesha multiple Availability Zones (AZs) mein deploy hota hai (minimum 2 AZs zaroori hain). AWS ka DNS mechanism client ke geographical location aur AZ ki health ke hisab se ALB ke dynamic Public IP Addresses ki ek list return karta hai.
- Client ka browser un IPs mein se ek IP address ko select kar leta hai. Aur request behjta hai.

<br>

**Stage 2: Connection Establishment & Security (Network Layer)**:

IP address milne ke baad, client directly ALB ke public-facing endpoints ke saath network level par communicate karta hai.
- **TCP Three-Way Handshake**: Client ka browser select kiye gaye ALB IP ke saath ek TCP connection initiate karta hai.
  - Client bhejta hai **SYN** packet.
  - ALB respond karta hai **SYN-ACK** packet ke saath.
  - Client bhejta hai **ACK** packet. Ab connection establish ho jata hai.
  - Iska faida yeh hai ki backend servers direct internet ke exposure mein nahi aate, jisse security badhti hai.
 
- **Port Verification**: ALB unhi ports par request accept karega jo uske Listeners mein configured hain (jaise HTTP ke liye port 80 aur HTTPS ke liye port 443). Agar koi client kisi unconfigured port par connection banayega, toh ALB connection drop kar dega.
- **AWS Shield (DDoS Protection)**: Request jaise hi ALB ke network border par aati hai, AWS Shield Standard automatically active rehta hai. Ye Layer 3 (IP layer) aur Layer 4 (TCP layer) ke SYN floods ya generic DDoS attacks ko yahan hi block kar deta hai taaki aapka load balancer crash na ho.

<br>

**Stage 3: SSL/TLS Offloading & Termination**:

Agar user ne ```https://``` use kiya hai, toh encryption aur security checks isi stage par handle hote hain.

- **SSL/TLS Handshake**: Application Load Balancer, client ke saath secure connection negotiate karta hai. Client aur ALB aapas mein cryptographic algorithms (Ciphers) aur TLS version (jaise TLS 1.2 ya TLS 1.3) agree karte hain.
- **Certificate Matching (SNI)**: Agar aapne ek hi ALB par multiple domains setup kiye hain, toh ALB Server Name Indication (SNI) ka use karta hai. Wo dekhta hai ki client kis domain (```https://example.com```) ke liye aaya hai aur AWS Certificate Manager (ACM) se wahi exact SSL certificate client ko server identity prove karne ke liye banta hai.
- **Decryption (Offloading)**: SSL Handshake hone ke baad ab ALB yahan par poore traffic ko decrypt karta hai. Iska sabse bada fayda ye hai ki aapke backend EC2 servers ko decryption ka heavy CPU-intensive kaam nahi karna padta. ALB saara encrypted traffic khud jhelta hai aur andar plain text data nikalta hai. Jab ALB SSL Handshake hone ke baad traffic ko decrypt karta hai to is process ko SSL Termination kehte hain.

<br>

**Stage 4: Layer 7 Request Parsing (Application Layer)**:

ALB ek intelligent Layer 7 load balancer hai, jiska matlab hai ki ye sirf network packets nahi dekhta, balki pure HTTP request ke content ko samajhta hai.

Yeh request ke ander check karta hai:
- **HTTP Method**: Reques ka HTTP Method kya hai. Jaise, GET, POST, PUT, DELETE, etc.
- **Host Header**: Kis exact website ke liye request hai (e.g., ```://example.com``` ya ```://example.com```).
- **URL Path**: User url ke konse route ko access karna chta hai (e.g., ```/images/logo.png``` ya ```/api/payment```).
- **Query Parameters**: URL ke peeche lagne wale variables (e.g., ```?user_id=50&status=active```).
- **HTTP Headers**: User-Agent, Cookies, Custom Headers.

**AWS WAF Integration (Optional)**: Agar aapne ALB ke saath AWS Web Application Firewall (WAF) joda hai, toh is step par WAF pure request content ko scan karega. Agar use SQL Injection, Cross-Site Scripting (XSS), ya kisi blacklisted IP ka pata chalta hai, toh ALB request ko aage bhejne ke bajay wahin se HTTP 403 Forbidden error return kar dega.

<br>

**Stage 5: Listener Rules & Routing Evaluation**:

Request ka data dekhne ke baad, ALB apne andar save kiye gaye Listener Rules ko upar se niche ek-ek karke check karta hai. Jaise hi koi rule match hota hai, ALB decide kar leta hai ki is request ko kis Target Group (servers ka group) ke paas bhejna hai.

Request parse hone ke baad, ALB ye decide karta hai ki is request ko backend ke kis **Target Group** (servers ka group) mein bhejna hai. Iske liye wo Listener Rules ko process karta hai.
- ALB ke listener rules ek order (Priority number) mein set hote hain. Rule 1, Rule 2, Rule 3... is tarah se ALB in rules ki sequential checking karta hai.
- ALB incoming request ke data ko rules ki conditions se match karta hai:
  - Example: Agar rule hai ki ```/profile``` wali requests ko ```User-Servers-Group``` ke paas bhejo, toh ALB us group ko select kar leta hai.
  - **Host-based routin**g: Agar host ```https://example.com``` hai, toh ```Blog``` Target Group ko bhejo.
  - **Path-based routing**: Agar url ke end mein path ```/v2/*``` hai, toh ```New App``` Target Group ko bhejo.
  - **Http-header routing**: Agar browser header ```User-Agent: Mobile``` hai, toh Mobile Target Group ko bhejo.
 
Jaise hi pehla rule match hota hai, ALB wahi ruk jata hai aur us rule mein defined Action ko trigger karta hai. Actions teen tarah ke ho sakte hain:
- **Forward**: Request ko kisi Target Group mein bhejna (Sabse common).
- **Redirect**: User ko kisi dusre URL par redirect karna (e.g., HTTP se HTTPS par shift karna, HTTP 301/302).
- **Fixed Response**: Backend server par jaye bina khud hi koi custom response code aur message dena (e.g., Maintenance page ke liye HTTP 503 return karna).

Agar upar ka koi bhi rule match nahi hota, toh sabse niche ek Default Rule hota hai. Request automatically default Target Group par chali jaati hai.

<br>

**Stage 6: Target Group Execution & Health Filtering**

Target Group ek collection hota hai aapke backend resources ka. Request yahan aane ke baad target ka selection shuru hota hai. Matlab ek backend resource select kiya jata hai request ko us server par bhejne ke liye.

**Target Type Identification**: Target group mein alag-alag tarah ke targets register ho sakte hain:
- **Instance ID**: Direct EC2 instances.
- **IP Address**: Pods (EKS), Containers (ECS), ya on-premises servers (via VPN).
- **Lambda Function**: Serverless code execution.

ALB background mein har target par continuous interval (e.g., har 30 seconds mein) par health checks chala raha hota hai (jaise ```/healthcheck``` path par HTTP 200 OK check karna). Jo server response nahi deta, use Unhealthy mark kiya jata hai. Is step par, ALB unhealthy servers ko selection pool se bahar nikal deta hai taaki user ko error na mile.

**Load Balancing Algorithm Application**: Pool mein bache healthy targets mein se ek target chunne ke liye do algorithms ka use hota hai:
- **Round Robin (Default)**: Requests ko har server par barabar baantta hai (Server A -> Server B -> Server C -> Server A).
- **Least Outstanding Requests**: ALB dekhta hai ki kis server par is waqt sabse kam active HTTP requests chal rahi hain, aur nayi request usi ko bhejta hai. Ye un applications ke liye best hai jahan kuch requests process hone mein zyada time leti hain.

Sticky Sessions (Session Affinity): Agar aapne Target Group mein stickiness enable ki hai, toh ALB check karta hai ki kya request ke saath ALB ki apni ek cookie (AWSALB) aayi hai. Agar aayi hai, toh routing algorithm ko bypass karke request ko usi exact purane server par bhej diya jata hai jahan user pehle connected tha.

<br>

**Stage 7: Request Modification & Backend Forwarding**:

Target select hone ke baad, ALB us server ke saath baat karne ki taiyari karta hai.
- ALB backend server ke private IP aur specified port ke saath ek naya TCP connection banata hai. ALB aksar backend servers ke saath connections ko open rakhta hai (Keep-Alive), taaki har request ke liye naya TCP handshake na karna pade. Isse latency bohot kam ho jaati hai.
  
- HTTP Header Injection: Kyunki ALB ek proxy ki tarah kaam kar raha hai, backend server ko aisa lagega ki request ALB se aayi hai, na ki asli user se. Asli details server tak pahunchane ke liye ALB request headers mein badlav karta hai:
  - ```X-Forwarded-For```: Isme user ka asli Public IP address append kiya jata hai.
  - ```X-Forwarded-Proto```: Server ko batata hai ki user https se aaya tha, chahe ALB se server tak ka traffic ab plain http ho.
  - ```X-Forwarded-Port```: Original port (e.g., 443) ki jankari deta hai.
 
- Modified request ko private network (VPC Subnets) ke zariye backend target ko deliver kar diya jata hai.

<br>

**Stage 8: Response Processing & Client Delivery**:

Ab Backend server apna kaam poora karke jab jawab (Response) taiyar karta hai, toh return journey shuruaat hoti hai.
- EC2 instance ya container request ko process karta hai (database query chalata hai, HTML/JSON generate karta hai) aur HTTP Response Code (jaise 200 OK, 404 Not Found, ya 500 Internal Server Error) ke saath response ALB ko wapas bhejta hai.
  
- ALB is transaction ki saari details (Client IP, Target IP, Request Path, Processing Time, Response Size, Status Code) apne paas note karta hai aur ise review ke liye AWS S3 buckets mein access logs ke roop mein bhej deta hai.
  
- Connection Re-Encryption (Optional): Agar backend target se ALB tak ka traffic bhi encrypted (https) tha, toh ALB use decrypt karega aur fir se user ke SSL certificates ke hisab se re-encrypt karega. Agar backend standard http tha, toh ALB plain text lekar use client ke secure TLS channel ke mutabiq encrypt karega.
  
- ALB response ko usi original TCP connection par push kar deta hai jo Stage 2 mein client ke browser ke saath khula tha. User ke browser par website load ho jaati hai.
