# LAB 3: Application Load Balancer Sticket Sessions

Is Lab mein hum Application Load Balancer ke ek feature Sticky Session ke baare mein dekhenge. Phle samjhenge hota kya hai, fir uski LAB dekhenge.

<br>

AWS Application Load Balancer (ALB) mein Sticky Sessions (ya Target Group Stickiness) ek aisa concept hai jo yeh ensure karta hai ki ek specific user/client se aane wali saari requests hamesha usi ek specific backend EC2 server par jayein jahan uski pehli request gayi thi.

<br>
<br>

### Stickiness Ki Zaroorat Kyu Hoti Hai?

Pichli lab mein jab aapne website ko refresh kiya, toh traffic kabhi Server A par ja raha tha aur kabhi Server B par (Round-Robin algorithm ki wajah se).

Lekin real-world applications (jaise E-commerce websites) mein user jab login karta hai, toh uski login state (Session Data, Shopping Cart items) usi server ki memory (RAM) mein save ho jati hai jahan woh connect hua tha.
- Agar Stickiness on nahi hogi, toh user ne pehle click par login kiya (Server A par).
- Jab usne dusra click kiya ("Add to Cart"), toh ALB ne request bhej di Server B par.
- Server B ke paas us user ka koi login record nahi hai, isliye website user ko bolega: "Firse login karo!"

**Solution**: Sticky Sessions open karne par ALB user ke browser mein ek unique Cookie save kar deta hai. Agli baar jab bhi user koi naya click karta hai, ALB us cookie ko dekhkar pehchan jata hai aur request ko chupchaap usi purane server par bhejta hai.

<br>
<br>

## Sticky Sessions Lab Scenario Overview

**Goal**: ALB par Stickiness enable karna aur verify karna ki refresh karne par bhi traffic ek hi server par stick rehta hai.

**Prerequisite**: Aapka purana Custom VPC setup jisme ```scratch-web-alb``` aur Target Group (```backend```) chal rahe hain.

<br>

### Step 1: Target Group Mein Stickiness Enable Karna

Stickiness ki settings hamesha **Target Group** ke level par configure hoti hain, Load Balancer ke level par nahi.
- AWS Console mein search bar mein **EC2** type karke dashboard par jayein
- Left-hand side menu ko scroll karein aur Load Balancing ke andar **Target Groups** par click karein.
- Apni purani lab wale Target Group (```backend```) ko select karein (uske naam par click karein).
- Andar aapko kaafi saare tabs dikhenge. Right side mein dekhein, aapko ek tab milega jiska naam hai **Attributes**. Us par click karein.
- Attributes section ke right side mein **Edit** button par click karein.
- Scroll karke neeche jayein, wahan aapko **Target selection configuration** ya **Stickiness** ka option dikhega:
  - **Stickiness Checkbox**: Ise **Enable** (tick) kar dein.
  - **Stickiness type**: Isko ```Load balancer generated cookie``` select karein (Yeh AWS ka apna default aur sabsé aasan tareeqa hai).
  - **Cookie expiration period**: Ise 1 minute ya 60 seconds set kar dein (Testing ke liye kam time rakhna best hota hai taaki hum iska expiry test kar sakein. Production mein yeh aamtaur par 1 din ya 1 ghanta hota hai).
- Sabse neeche jaakar **Save changes** button par click kar dein.

<br>

### Step 2: Live Production Verification (Testing The Stickiness)

Ab hum verify karenge ki kya hamara Load Balancer sach mein ek hi server par traffic rok raha hai.
- Apne Load Balancer (```scratch-web-alb```) ka DNS Name / URL copy karein.
- Apne browser mein ek naya **Incognito (Private) Window** kholein.
- URL ke aage manually ```http://``` lagakar hit karein (Jaise: ```http://amazonaws.com```).
- **First Output Check**: Maan lijiye screen par likha aaya: ```Hello from Web Server A```.
- **The Magic Test (Refresh Cycle)**: Ab apne keyboard par **F5** ya browser ka **Refresh** button lagatar 5 se 10 baar dabaayein.
  - Pehle kya ho raha tha? Har refresh par output A se B aur B se A badal raha tha.
  - Ab kya hoga? Aap chahe jitni baar refresh karein, screen par lagatar sirf aur sirf ```Hello from Scratch Private Server A``` hi dikhta rahega. Load Balancer ne aapko Server A ke saath "stick" (baandh) diya hai.
 
<br>

### Step 3: Cookie Expiry Aur Core Mechanism Test

Kyunki humne cookie ka duration sirf **1 minute** set kiya tha, chaliye dekhte hain ki timer khatam hone ke baad kya hota hai.
- Apne laptop/system par 1 minute (60 seconds) tak ka wait karein aur koi click na karein.
- Jaise hi 1 minute poora ho jaye, browser par firse **Refresh** dabayein.
- **Result**: Is baar aapka response change hokar ```Hello from Scratch Private Server B``` ho sakta hai!
- **Kyu hua aisa?** Kyunki 1 minute khatam hote hi aapke browser se woh sticky cookie expire ho gayi. ALB ne aapko ek naya user samjha aur Round-Robin ke mutabik request ko dusre available server (Server B) par bhej diya. Aur ab agle 1 minute tak aap Server B ke saath stick rahenge.

### Sticky Sessions Ke Nuksaan (Production Interview Question)

Interviewers aamtaur par poochte hain ki agar Stickiness itni acchi hai toh isko hamesha on kyu nahi rakhte?
- **Imbalanced Load**: Agar koi ek bada corporate user (jaise ek office jahan 500 log ek hi public IP se website chala rahe hain) aapki site par aata hai, toh ALB un sabhi 500 logon ko kisi ek hi server (Server A) par bhej dega. Isse Server A par load bohot zyada badh jayega aur Server B khali baitha rahega.
- Isliye modern stateless applications mein stickiness band karke session data ko ek central database jaise **Amazon ElastiCache (Redis)** ya **DynamoDB** mein save kiya jata hai.
