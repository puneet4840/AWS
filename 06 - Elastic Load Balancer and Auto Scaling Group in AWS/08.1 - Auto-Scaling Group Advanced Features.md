# Auto-Scaling Group Advanced Feature


### Auto-Scaling Group Termination Policy

AWS Auto Scaling Group (ASG) ke andar Termination Policy ek bohot hi important setting hoti hai. Yeh policy ASG ko yeh batati hai ki jab traffic kam hoga (Scale-In process ke dauran), toh chal rahe multiple servers mein se sabse pehle kaun sa server delete (terminate) karna hai.

Agar aap koi policy select nahi karte, toh ASG apni Default Termination Policy ka use karta hai.

<br>

**1. Default Termination Policy Kaise Kaam Karti Hai?**:

AWS ki default policy ko is tarah design kiya gaya hai ki aapka application hamesha safe rahe aur aapke servers alag-alag data centers (Availability Zones) mein barabar distribut reh sakein. Yeh policy niche diye gaye steps ke hisab se decision leti hai:
- **Step 1 (Availability Zones Balance)**: ASG sabse pehle yeh dekhta hai ki kis Availability Zone (AZ) mein sabse zyada instances chal rahe hain. Yeh hamesha us AZ ko chunta hai jisme servers ki ginti zyada hai, taaki baki bache servers sabhi data centers mein barabar divide rahein.
- **Step 2 (Launch Template Comparison)**: Agar kisi AZ mein extra servers hain, toh ASG check karta hai ki kya unme se koi server purane Launch Template ya Launch Configuration se bana hai. Agar aapne haal hi mein application ka naya version update kiya hai, toh ASG sabse pehle purane version wale server ko delete karega.
- **Step 3 (Next Billing Hour)**: Agar sabhi servers ek hi template se bane hain, toh ASG yeh dekhta hai ki kaun sa server apne agle billing hour (ya cost cycle) ke sabse kareeb hai. Isse AWS bill mein paise bachte hain.
- **Step 4 (Oldest Instance)**: Agar upar ke saare rules tie ho jate hain, toh ASG us server ko chunta hai jo sabse purana hai (Oldest Instance) aur use terminate kar deta hai.


<br>

**2. Custom Termination Policies**:

Aap apni business requirement ke hisab se default policy ko badal kar niche di gayi custom policies mein se koi bhi chun sakte hain:

- **OldestInstance (Sabse Purana Server)**: Is policy ko select karne par ASG hamesha us server ko delete karega jo sabse pehle launch hua tha. Yeh tab kaam aata hai jab aap chahte hain ki aapke system mein purane servers rotate hote rahein aur naye fresh servers chalte rahein.
- **NewestInstance (Sabse Naya Server)**: Yeh policy sabse naye launch huye server ko pehle delete karti hai. Yeh tab use hoti hai jab aap kisi short-term testing ya temporary high traffic ko handle kar rahe hote hain aur aap chahte hain ki jo server abhi-abhi aaya hai, wahi pehle chala jaye.
- **OldestLaunchTemplate (Purana Blueprint)**: Agar aapne apne application ka naya code deploy kiya hai aur naya Launch Template banaya hai, toh yeh policy un servers ko dhoondkar pehle delete karegii jo abhi bhi purane template par chal rahe hain. Isse aapka naya code automatic sabhi servers par live ho jata hai.
- **ClosestToNextInstanceHour (Billing Cycle Ke Paas)**: Yeh policy billing ke hisab se kaam karti hai. Yeh us server ko pehle delete karti hai jiska agla hourly payment shuru hone wala hota hai, jisse aapko us server ka extra charge na dena pade.

<br>

**3. Instance Protection (Kisi Khaas Server Ko Delete Hone Se Bachana)**:

Kabhi-kabhi aisi situation aati hai jahan ASG kisi server ko termination policy ke hisab se delete karne jata hai, lekin aap nahi chahte ki woh server delete ho. Jaise ki:
- Us server par koi bada backend job chal raha hai jo adha poora ho chuka hai.
- Us server par kisi user ka important live session chal raha hai.

Is cheez ko handle karne ke liye AWS mein Scale-In Protection ka option hota hai. Aap kisi bhi specific EC2 instance par is feature ko "On" kar sakte hain. Jab tak Scale-In Protection active rahega, ASG us server ko chhoo bhi nahi sakta, bhale hi aapki termination policy kuch bhi keh rahi ho.

<br>

**4. Custom Termination Policy Kaise Set Karte Hain?**

Aap AWS Management Console mein ja kar apne Auto Scaling Group ki settings ko edit kar sakte hain. Wahan aapko ek dropdown menu milega jahan se aap upar batayi gayi kisi bhi policy ko select kar sakte hain. Aap chahein toh multiple policies ko ek sequence (order) mein bhi daal sakte hain (jaise: Pehle OldestLaunchTemplate check karo, agar sab same hain toh OldestInstance delete karo).
