# Auto-Scaling Group Advanced Feature


### Auto-Scaling Group Termination Policy

AWS Auto Scaling Group (ASG) ke andar Termination Policy ek bohot hi important setting hoti hai. Yeh policy ASG ko yeh batati hai ki jab traffic kam hoga (Scale-In process ke dauran), toh chal rahe multiple servers mein se sabse pehle kaun sa server delete (terminate) karna hai.

Agar aap koi policy select nahi karte, toh ASG apni Default Termination Policy ka use karta hai.

<br>

**1. Default Termination Policy Kaise Kaam Karti Hai?**:

AWS ki default policy ko is tarah design kiya gaya hai ki aapka application hamesha safe rahe aur aapke servers alag-alag data centers (Availability Zones) mein barabar distribut reh sakein. Yeh policy niche diye gaye steps ke hisab se decision leti hai:
- **Step 1 (Availability Zones Balance)**: ASG sabse pehle yeh dekhta hai ki kis Availability Zone (AZ) mein sabse zyada instances chal rahe hain. Yeh hamesha us AZ ko chunta hai jisme servers ki ginti zyada hai, taaki baki bache servers sabhi data centers mein barabar divide rahein.
- **Step 2 (Launch Template Comparison)**: Agar kisi AZ mein extra servers hain, toh ASG check karta hai ki kya unme se koi server purane Launch Template ya Launch Configuration se bana hai. Agar aapne haal hi mein application ka naya version update kiya hai, toh ASG sabse pehle purane version wale server ko delete karega.
- **Step 3 (Next Billing Hour)**: Agar sabhi servers ek hi template se bane hain, toh ASG yeh dekhta hai ki kaun sa server apne agle billing hour (ya cost cycle) ke sabse kareeb hai. Isse AWS bill mein paise bachte hain.
- **Step 4 (Oldest Instance)**: Agar upar ke saare rules tie ho jate hain, toh ASG us server ko chunta hai jo sabse purana hai (Oldest Instance) aur use terminate kar deta hai.


<br>

**2. Custom Termination Policies**:

Aap apni business requirement ke hisab se default policy ko badal kar niche di gayi custom policies mein se koi bhi chun sakte hain:

- **OldestInstance (Sabse Purana Server)**: Is policy ko select karne par ASG hamesha us server ko delete karega jo sabse pehle launch hua tha. Yeh tab kaam aata hai jab aap chahte hain ki aapke system mein purane servers rotate hote rahein aur naye fresh servers chalte rahein.
- **NewestInstance (Sabse Naya Server)**: Yeh policy sabse naye launch huye server ko pehle delete karti hai. Yeh tab use hoti hai jab aap kisi short-term testing ya temporary high traffic ko handle kar rahe hote hain aur aap chahte hain ki jo server abhi-abhi aaya hai, wahi pehle chala jaye.
- **OldestLaunchTemplate (Purana Blueprint)**: Agar aapne apne application ka naya code deploy kiya hai aur naya Launch Template banaya hai, toh yeh policy un servers ko dhoondkar pehle delete karegii jo abhi bhi purane template par chal rahe hain. Isse aapka naya code automatic sabhi servers par live ho jata hai.
- **ClosestToNextInstanceHour (Billing Cycle Ke Paas)**: Yeh policy billing ke hisab se kaam karti hai. Yeh us server ko pehle delete karti hai jiska agla hourly payment shuru hone wala hota hai, jisse aapko us server ka extra charge na dena pade.

<br>

**3. Instance Protection (Kisi Khaas Server Ko Delete Hone Se Bachana)**:

Kabhi-kabhi aisi situation aati hai jahan ASG kisi server ko termination policy ke hisab se delete karne jata hai, lekin aap nahi chahte ki woh server delete ho. Jaise ki:
- Us server par koi bada backend job chal raha hai jo adha poora ho chuka hai.
- Us server par kisi user ka important live session chal raha hai.

Is cheez ko handle karne ke liye AWS mein Scale-In Protection ka option hota hai. Aap kisi bhi specific EC2 instance par is feature ko "On" kar sakte hain. Jab tak Scale-In Protection active rahega, ASG us server ko chhoo bhi nahi sakta, bhale hi aapki termination policy kuch bhi keh rahi ho.

<br>

**4. Custom Termination Policy Kaise Set Karte Hain?**

Aap AWS Management Console mein ja kar apne Auto Scaling Group ki settings ko edit kar sakte hain. Wahan aapko ek dropdown menu milega jahan se aap upar batayi gayi kisi bhi policy ko select kar sakte hain. Aap chahein toh multiple policies ko ek sequence (order) mein bhi daal sakte hain (jaise: Pehle OldestLaunchTemplate check karo, agar sab same hain toh OldestInstance delete karo).

### Cool Down Period

AWS Auto Scaling Group (ASG) ke andar Cooldown Period ek aisi setting hoti hai jo aapke system ko stable rakhne aur paise ki barbadi se bachane ke liye bohot zaruri hai. Jab ASG koi scaling activity karta hai (jaise koi naya server add karna ya purana server delete karna), toh uske turant baad woh ek specific time ke liye shant baith jata hai. Usi silent duration ko hum Cooldown Period kehte hain.

<br>

**1. Cooldown Period Kyu Zaroori Hai? (The Problem)**:

Maan lijiye aapki website par achanak traffic badha aur CPU usage 80% ho gaya. ASG ne scaling policy ke hisab se turant ek naya EC2 instance launch karne ka order de diya.Ab naya server launch hone, boot hone, operating system start karne aur aapke application code ko load karne mein kam se kam 2 se 3 minutes ka time lagta hai. Is 2-3 minutes ke dauran, purane servers par abhi bhi load high rahega kyunki naya server abhi traffic lene ke liye taiyar nahi hai.Agar cooldown period nahi hoga, toh ASG har 1 minute baad check karega aur dekhega ki "Are, CPU toh abhi bhi 80% hai!" aur woh fir se ek aur naya server launch kar dega. Aisa karte-karte woh zaroorat se zyada servers launch kar dega jisse aapka AWS bill faltu mein badh jayega.

<br>

**2. Yeh Kaise Kaam Karta Hai?**

Cooldown period is problem ko aise solve karta hai:
- **Step 1**: Traffic badha aur ASG ne ek naya server launch kiya.
- **Step 2**: Jaise hi naya server launch hota hai, ASG ka Cooldown Timer shuru ho jata hai (default time 300 seconds yaani 5 minutes hota hai).
- **Step 3**: Is 5 minutes ke dauran ASG baki saare scaling alarms ko ignore kar deta hai. Woh koi naya server launch nahi karega, bhale hi CPU usage tab tak 90% kyu na chala jaye.
- **Step 4**: Is gap mein naya server poori tarah se ready ho jata hai, load balancer se connect hota hai aur traffic ko share karna shuru kar deta hai.
- **Step 5**: Jaise hi naya server load handle karta hai, CPU usage automatic niche aa jata hai. Jab 5 minutes khatam hote hain, tab tak system normal ho chuka hota hai aur koi extra faltu server launch nahi hota.

<br>

**3. Cooldown Periods Ke Types**:

AWS mein teen tarah ke cooldown periods hote hain jo alag-alag situations mein kaam aate hain:

**A. Default Cooldown Period**

Yeh poore Auto Scaling Group par apply hota hai. Jab bhi aap Simple Scaling Policy ka use karte hain, toh har scaling activity (chahe server add ho raha ho ya remove ho raha ho) ke baad yeh default timer shuru ho jata hai. AWS isko standard 300 seconds par set rakhta hai, lekin aap isko kam ya zyada kar sakte hain.

**B. Scaling-Specific Cooldown (Policy Level)**:

Agar aap chahte hain ki server add karte samay (Scale-out) cooldown period 2 minutes ka rahe, lekin server delete karte samay (Scale-in) cooldown period 5 minutes ka ho, toh aap alag-alag scaling policies ke andar apna custom cooldown time set kar sakte hain. Yeh default cooldown ko override kar deta hai.

**C. Warm-up Period (Target Tracking Policies Ke Liye)**:

Agar aap Target Tracking Scaling Policy (jaise: Average CPU hamesha 70% rahe) use kar rahe hain, toh AWS cooldown ki jagah Instance Warm-up time ka use karta hai. Isme aap ASG ko batate hain ki mere naye server ko poori tarah ready hone mein kitna time lagta hai. Jab tak naya server warm-up ho raha hota hai, ASG tab tak purane servers ke data ko measure nahi karta, jisse galat scaling decisions nahi hote.

<br>

**4. Isko Kitna Set Karna Chahiye?**

Cooldown period ka time is baat par depend karta hai ki aapka application server start hone mein kitna time leta hai:
- Agar aapka application ek chota Docker container hai jo 30 seconds mein chalu ho jata hai, toh aap cooldown period ko kam karke 60 ya 90 seconds kar sakte hain taaki scaling bohot fast ho.
- Agar aapka application bohot bada hai jisme bade softwres load hote hain aur use ready hone mein 4 minutes lagte hain, toh aapko cooldown period kam se kam 300 seconds (5 minutes) rakhna chahiye.

### Instance Refresh

AWS Auto Scaling Group (ASG) ke andar Instance Refresh ek aisa feature hai jo chal rahe servers (EC2 instances) ko bina kisi downtime ke naye version ya nayi settings ke sath automatic update karne mein help karta hai.

Maan lijiye aapne apne application ka naya code deploy karna hai ya server ka Operating System patch karna hai, toh aap poore group ke servers ko ek-ek karke safely replace karne ke liye Instance Refresh ka use karte hain.

<br>

**1. Instance Refresh Ki Zaroorat Kyu Hoti Hai?**

Normal dino mein jab traffic badhta hai toh ASG naye servers launch karta hai. Lekin agar aapko chal rahe maujooda servers ke andar koi badlav karna ho, jaise:
- Aapne naya Launch Template banaya hai jisme application ka naya code version (V2) hai.
- Aapne server ka size badalna hai (jaise t2.micro se t3.micro par jana hai).
- Aapne security patches ya Operating System (AMI) ko update kiya hai.

Agar aap manually ek-ek server ko delete karenge, toh isme bohot time lagega aur galti hone ka chance rahega. Instance Refresh is poore rolling update process ko automatic aur safe tarike se handle karta hai.

<br>

**2. Yeh Kaise Work Karta Hai? (Step-by-Step Flow)**:

Jab aap Instance Refresh trigger karte hain, toh ASG niche diye gaye tarike se rolling update shuru karta hai:
- Step 1 (Check Settings): ASG aapke naye Launch Template ko dekhta hai aur purane chal rahe servers ki list banata hai.
- Step 2 (Naye Servers Launch Karna): Yeh sabse pehle naye settings wale servers ko launch karta hai taaki website par capacity kam na ho.
- Step 3 (Target Group Registration): Naye servers ko Application Load Balancer (ALB) ke target group mein add kiya jata hai aur unka health check pass hone ka wait kiya jata hai.
- Step 4 (Purane Servers Delete Karna): Jaise hi naye servers active hote hain, ASG purane servers se traffic hatana (Connection Draining) shuru karta hai aur unhe terminate (delete) kar deta hai.
- Step 5 (Rolling Process): Yeh process tab tak chalta rehta hai jab tak group ke saare purane servers naye servers se replace nahi ho jate.

<br>

**3. Iske Main Configuration Parameters**:

Instance Refresh ko customize karne ke liye AWS aapko do sabse main settings deta hai:

**A. Minimum Healthy Percentage**:

Yeh setting aapke application ko downtime se bachati hai. Isme aap ASG ko batate hain ki poore refresh process ke dauran kam se kam kitne percent servers hamesha healthy aur running state mein hone chahiye.
- Example: Agar aapke paas 4 servers chal rahe hain aur aapne Minimum Healthy Percentage 50% set kiya hai, toh ASG ek baar mein maximum 2 servers ko terminate kar sakta hai aur baaki 2 servers live rahenge.
- Agar aapne ise 100% set kiya, toh ASG pehle 4 naye servers launch karega, aur jab woh ready ho jayenge tabhi purane 4 servers ko delete karega.

**B. Instance Warm-up Time**:

Yeh woh time duration hoti hai jo ASG naye server ke launch hone ke baad wait karta hai. Jab tak naya server poori tarah boot hokar ready nahi ho jata aur uska warm-up time khatam nahi hota, ASG agle purane server ko replace karne ka process shuru nahi karta. Isse system ko stable hone ka poora time milta hai.

<br>

**4. Skip Matching (Paise Aur Time Bachana)**:

Instance Refresh ke andar ek bohot acha feature hota hai jise Skip Matching kehte hain. Agar aapke ASG ke andar kuch servers pehle se hi naye Launch Template par chal rahe hain (kyunki woh haal hi mein kisi scaling alarm ki wajah se naye launch huye the), toh ASG unhe chhooega bhi nahi. Yeh sirf unhi servers ko target karega jo sach mein purane template par chal rahe hain, jisse aapka deployment jaldi poora ho jata hai.

<br>

**5. Checkpoints (Controlled Deployment)**:

Agar aap apne deployment ko thoda aur control karna chahte hain, toh aap Checkpoints set kar sakte hain. Iska matlab hai ki aap ASG ko bol sakte hain ki "Pehle sirf 25% servers ko update karo, fir 15 minutes ke liye ruk jao".

Is break ke dauran aap check kar sakte hain ki naye code mein koi bug toh nahi hai. Agar sab sahi lagta hai, toh refresh process aage badhta hai. Agar koi issue milta hai, toh aap process ko turant Rollback (cancel) kar sakte hain taaki bache huye baaki servers safe rahein.

### Life Cycle Hooks

AWS Auto Scaling Group (ASG) ke andar Lifecycle Hooks ek bohot hi powerful aur advanced feature hai. Yeh aapko kisi server ke launch hone ke turant baad, ya fir uske delete (terminate) hone ke theek pehle, beech mein rok kar koi custom kaam karne ki ijaajat deta hai.

Normal halat mein, ASG turant server ko group mein jod deta hai ya delete kar deta hai. Lekin Lifecycle Hooks ki madad se aap us process ko beech mein hi freeze (pause) kar sakte hain.

<br>

**1. Lifecycle Hooks Ki Zaroorat Kyu Hoti Hai?**

Maan lijiye aapka naya server launch toh ho gaya, lekin uske upar abhi kuch zaroori kaam baaki hain, jaise:
- Website ka sabse naya code GitHub ya S3 bucket se download karna.
- Kisi software ka license key register karna.
- Server ke andar zaroori security updates install karna.

Agar bina yeh sab kiye server direct load balancer ke paas chala gaya, toh users ko adhura application dikhega aur error aane lagegi.

Similarly, jab koi server delete ho raha ho, toh aap chahenge ki:
- Us server ke andar save huye important logs ko S3 bucket par backup kiya ja sake.
- Chal rahe kisi database connection ko safely close kiya ja sake.

Inhi sab kaamon ke liye Lifecycle Hooks ka use kiya jata hai.

<br>

**2. Yeh Do Main Stages Par Kaam Karta Hai**:

ASG ke paas do tarah ke hooks hote hain jahan aap action le sakte hain:

- A. Instance Launching Hook (autoscaling:EC2_INSTANCE_LAUNCHING)
  - Kab trigger hota hai: Jab ASG ek naya server launch karta hai.
  - Kya hota hai: Server launch hote hi ASG use ek Wait State (intezar ki halat) mein daal deta hai. Server load balancer se connect nahi hota.
  - Aap kya karte hain: Is pause ke dauran aapka koi script ya AWS Lambda function server par naya code install karta hai ya configuration poori karta hai.
  - Aage kya hota hai: Jab aapka kaam poora ho jata hai, toh aap ASG ko ek signal bhejte hain (CONTINUE). Iske baad ASG server ko active maante huye load balancer ke sath attach kar deta hai.
 
- B. Instance Terminating Hook (autoscaling:EC2_INSTANCE_TERMINATING)
  - Kab trigger hota hai: Jab scale-in process ke dauran ASG kisi server ko delete karne wala hota hai.
  - Kya hota hai: ASG server ko load balancer se toh hata deta hai, lekin use turant delete nahi karta. Use fir se ek Wait State mein daal deta hai.
  - Aap kya karte hain: Is dauran aapka script server ke important data ya log files ko uthakar kisi safe jagah (jaise Amazon S3) par upload kar deta hai.
  - Aage kya hota hai: Kaam poora hone par aap ASG ko signal bhejte hain, aur ASG us server ko permanently terminate (delete) kar deta hai.
 
<br>

**3. Heartbeat Timeout Aur Result (Sabse Main Settings)**:

Lifecycle Hook ko control karne ke liye do settings bohot zaroori hoti hain:

- **Heartbeat Timeout**: Yeh woh time duration hoti hai jo aap ASG ko batate hain ki "Mera kaam poora hone ke liye maximum 5 minutes (ya 300 seconds) tak ka wait karna". Agar is time ke andar aapka script kaam khatam karke signal nahi bhej pata, toh ASG khud hi action le leta hai. Aap is timeout ko zaroorat ke hisab se badha bhi sakte hain.
  
- **Default Result**: Agar timeout khatam ho jaye aur aapka script koi signal na bhej paye, toh ASG kya kare? Iske do options hote hain:
  - CONTINUE: ASG samjhega ki sab theek hai aur process ko aage badha dega.
  - ABANDON: ASG samjhega ki kaam fail ho gaya hai. Launching ke waqt yeh us adhoore server ko delete karke ek naya fresh server khada karne ki koshish karega.
 
<br>

**4. Yeh Backend Mein Kaise Work Karta Hai?**

Jab bhi koi Lifecycle Hook trigger hota hai, toh ASG chupchaap baithne ke sath-sath ek notification bhejta hai. Yeh notification aap Amazon SNS (Simple Notification Service) ya Amazon EventBridge par bhej sakte hain.

Wahan se ek AWS Lambda function trigger hota hai, jo aapke server ke andar ja kar zaroori scripts chalata hai. Jaise hi script khatam hota hai, Lambda function AWS API ke zariye ASG ko CompleteLifecycleAction ka command bhejta hai, aur ASG apna aage ka kaam shuru kar deta hai.
