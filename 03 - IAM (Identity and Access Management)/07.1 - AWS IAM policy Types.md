# AWS IAM Policy Types

<br>

### Sabse pehle problem samajhte hain

Maan lo tumhari company mein ek Developer hai.

Uska naam hai **Rahul**.

Rahul ko S3 Bucket access karna hai.

Ab sawal hai.

Permission kisko di jaye?

Rahul ko?

Ya 

S3 Bucket ko bola jaye ki Rahul ko andar aane dena?

Sunne mein dono same lagte hain. Lekin AWS mein ye dono alag approaches hain. Isi wajah se AWS mein do tarah ki policies hoti hain.

- Identity-Based Policy.
- Resource-Based Policy.

### Identity-Based Policy

Identity-Based Policy ek IAM Policy hoti hai jo IAM Identiyt jaise IAM User, IAM Group ya IAM Role ke saath attach hoti hai aur batati hai ki IAM User, IAM Group aur IAM Role kis AWS Resource par kya action perform kar sakte hai.

<br>

Sabse pehle Identity ka matlab samajhte hain.

Identity matlab hai jo AWS mein jo login karta hai.

Jaise.
- IAM User
- IAM Group
- IAM Role

Yahi identities hain.

Ab agar tum Policy in identities ke saath attach karte ho. To usse bolte hain **Identity-Based Policy**.

**Example-1**:

Rahul ek IAM User hai.

Tum uske saath Policy attach karte ho.
```
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "*"
}
```
Ruhul ko S3 se get object ki policy attach kardi. Ab Rahul S3 se file download kar sakta hai.

Yahan permission Rahul ke paas hai. Bucket ke paas nahi.

<br>

**Example-2**:

Company mein 100 Developers hain. Sabko EC2 Start aur Stop karna hai.

Tum kya karoge?

Har EC2 ke upar permission likhoge?

Nahi.

Ek Developers Group banaoge.

Aur Group ke saath Policy attach kar doge.
```
Developers Group

↓

EC2 Start

EC2 Stop
```

Ab Group ke saare Developers ko permission mil gayi. Ye Identity-Based Policy hai.

<br>
<br>

**Identity-Based Policy kis-kis par attach hoti hai?**

Ye sirf teen identities ke saath attach hoti hai.
- IAM User
- IAM Group
- IAM Role

Aur kisi ke saath nahi.

**IAM User**:

Admin direct Puneet ke IAM User account par jaakar ek policy attach kar deta hai.

<br>

Example:

Puneet naam ke IAM User ke saath ek policy attach ki gayi.

Policy allow karti hai:
- EC2 Start
- EC2 Stop

Ab jab Puneet EC2 instance start karega, AWS uski identity ke saath attached policy evaluate karega.

<br>

**IAM Group**:

Jaise aapki company mein 10 developers hain, toh aap un sabke alag-alag accounts par permissions lagane ke bajaye ek group bana dete hain jiska naam hota hai "Developers-Group". Is policy ko aap is group par attach kar dete ho.

Aap policy ko kisi ek single user par nahi lagate, balki poore group par attach kar dete hain.

Example:

Developer Group ke saath policy attach hai.

Policy allow karti hai:
- S3 Read
- CloudWatch Read

Rahul aur Amit Developer Group ke members hain. Rahul aur Amit ko ye permissions automatically mil jaayengi. Kyunki Group ki policy members inherit karte hain.

Yeh AWS ka ek Best Practice rule hai. Jab aap policy ko group par attach karte hain, toh us group ke andar jitne bhi users hote hain, un sabko woh permissions automatically mil jaati hain. Agar kal ko koi naya developer company join karta hai, toh aapko uske liye nayi policy nahi banani padegi. Aap bas use us group mein daal denge aur saari permissions use turant mil jayengi. Isse admin ka kaam bohot aasan ho jata hai.

<br>

**IAM Role**:

IAM Role kisi ek specific person ke liye nahi hota. Yeh ek tarah ki temporary identity (kuch samay ki pehchan) hoti hai jise koi bhi valid user ya AWS ki service adopt (use) kar sakti hai.

Aap policy ko Role ke sath attach karte hain. Aur jab bhi koi service ya user us role ko assume (dharn) karta hai, toh use temporary period ke liye woh permissions mil jaati hain.



Example:

AWS Services Ke Liye: Maan lijiye aapka ek EC2 Instance (Virtual Server) chal raha hai aur use S3 Bucket (Storage) se files download karni hain. EC2 ke paas apne khud ke password nahi hote. Toh aap ek Role banayenge, usme S3 read karne ke policy attach karenge, aur woh Role us EC2 server ko de denge.

Cross-Account Access Ke Liye: Agar aapki company ke do AWS accounts hain (Account A aur Account B). Account A ka user Account B mein kaam karna chahta hai, toh Account B mein ek Role banaya jayega jisme policy attached hogi, aur Account A ka user us role ko use karke kaam kar payega.

<br>
<br>

### Types of Identity-Based Policy:

Identity-Based policies further 2 types ki hoti hain:
- Managed Policies.
  - Customer Managed Policy.
  - AWS Managed Policy.
 
- Inline Policy.

<br>

**Managed Policy**:

Managed policy ka matlab hai ki in policies ko koi na koi manage karta hai, jaise in policy ko ya to AWS manage karta hai ya fir customer khud manage karta hai.

Managed Policy 2 types ki hoti hain:
- AWS Managed Policy.
- Customer Managed Policy.

**1. AWS Managed Policy**:

Yeh wo policies hain jo AWS ke engineers ne pehle se banakar aapke account mein di hui hain. AWS in policy ko pehle se create karke rakhta hai aur khud manage karta hai isliye inhe AWS managed policy kehte hain.

- Iska poora control AWS ke paas hota hai. Jab bhi AWS koi naya feature ya API launch karta hai, wo in policies ko background mein khud update kar deta hai.
- Aapko koi mehnat nahi karni padti. Agar aapko kisi ko admin banana hai, to aap chupchaap ```AdministratorAccess``` attach kar dete hain.
- Aap iske andar likhe gaye JSON code ko change ya edit nahi kar sakte. Yeh bilkul read-only hoti hain. Agar isme extra permissions hain jo aapko nahi chahiye, to aap unhe hata nahi sakte.

**2. Customer Managed Policy**:

Customer-Managed policy ko tum khud apne project ki requirement ke hisab se create karte ho. Matlab customer khud in policy ko khud ke hisab se create aur manage karta hai isliye inko customer-managed policy kehte hain.

- Iska 100% control aapke paas hota hai. Aap jab chahein iske andar ki permissions ko badal sakte hain.
- Security ka ek niyam hai jise "Principle of Least Privilege" kehte hain (yani user ko sirf utni hi permission do jitni kaam ke liye zaroori ho). Yeh niyam follow karne ke liye aapko Customer Managed Policies hi banani padti hain, kyunki AWS managed policies aksar bohot zyada broad (badi) hoti hain.


**Managed Policy Ke Fayde**:
- Centralized Control:
  - Maano aapki company mein 50 developers hain aur un sabko EC2 instances chalane ki permission chahiye. Agar aapne ek Managed Policy banayi jiska naam rakha ```Dev-EC2-Policy``` aur use un 50 users se attach kar diya.
    - Kal ko agar company ka niyam badalta hai aur developers ko S3 bucket ka access bhi dena padta hai, to aapko 50 users ke paas alag-alag jaane ki zaroori nahi hai.
    - Aap bas us ek ```Dev-EC2-Policy``` ko edit karke usme S3 ki permission jod denge. Wo permission automatically aur turant un sabhi 50 users par apply ho jayegi.
 
- Version Control (Back-up System):
  - Jab aap ek Customer Managed Policy ko edit karte hain, to AWS purane code ko delete nahi karta. Wo us policy ka ek naya Version bana deta hai (Jaise v1, v2, v3).
    - AWS ek policy ke maximum 5 versions ko store karke rakh sakta hai.
    - Iska fayda yeh hai ki agar aapne v2 version mein koi galat permission daal di jisse app crash ho gayi, to aap bina code badle, bas ek click se v1 version ko vapas "Default Version" set kar sakte hain. Yeh rollback facility inline policies mein nahi milti.
   
- Attach Limits:
  - Ek single managed policy ko aap apne AWS account ke andar hazaron users, groups, aur roles ke sath attach kar sakte hain. Iski koi broad limit nahi hai reuse karne ki. Wahi dusri taraf, ek single IAM user ya role ke upar aap maximum 20 Managed Policies attach kar sakte hain (is limit ko request karke badhaya bhi ja sakta hai).

**Best Practices**:
- Groups Ka Use Karein: Managed policy ko direct user se attach karne ki bajaye, ek IAM Group banayein (jaise Marketing-Group), policy ko us group se attach karein, aur users ko us group ka member bana dein.
- AWS Managed Se Shuru Karein, Custom Par Shift Karein: Jab project naya ho, to AWS managed policies use kar sakte hain taaki kaam jaldi ho. Par jaise hi production mein jayein, apni khud ki strict Customer Managed Policy banayein.
- Explicit Deny Samjhein: Agar kisi Managed policy mein kisi cheez ko ```Deny``` kar diya gaya hai, to chahe kisi dusri policy mein use kitna bhi ```Allow``` kiya gaya ho, wo block hi rahega. AWS mein ise Explicit Deny kehte hain. Isko allow karne ke liye apko deny policy ko remove karna hoga,tabi allow kaam karega.

<br>

**Inline Policy**:

Jab aap kisi specific IAM user, group ya role ke ander jake ek policy create karte hain, to use hi Inline Policy kehte hain.

- Jaise Managed Policy ka ek apna unique Amazon Resource Name (ARN) hota hai (jaise: ```arn:aws:iam::aws:policy/AmazonS3FullAccess```), Inline Policy ka aisa koi ARN nahi hota.
- Strictly Embedded: Yeh policy us IAM entity (user, group, ya role) ke andar hi embedded (dhansi hui) hoti hai. Yeh us user ka ek hissa (attribute) ban jati hai.
- 1-to-1 Relationship: Iska rishta hamesha 1-to-1 hota hai. Ek inline policy sirf ek hi user ya role ke liye banti hai. Aap ek hi inline policy ko 5 alag-alag users par apply nahi kar sakte. Agar un 5 users ko same permission chahiye, to aapko pancho ke andar ja kar alag-alag inline policy likhni padegi.

**Inline Policy Ka Lifecycle**:
- Agar aapne "Amit" naam ke ek IAM user ke andar ja kar ek inline policy banayi, aur 6 mahine baad aapne "Amit" ka AWS account se user delete kar diya, to uske andar ki inline policy bhi hamesha ke liye automatic delete ho jayegi.
- Managed policy ke case mein kya hota hai? Agar user delete ho jaye, to policy account mein padi rehti hai (ise orphan state kehte hain). Par inline policy kabhi akeli nahi reh sakti. User khatam, to policy bhi khatam.

**AWS Console Mein Yeh Kaise Dikhti Aur Banti Hai?**:

Jab aap AWS Management Console mein jaate hain, to iska workflow bilkul aapke bataye mutabik hota hai:
- Aap IAM Dashboard mein jaate hain.
- Aap Users (ya Roles/Groups) par click karte hain.
- Aap kisi ek user (jaise: ```Dev-User-Rahul```) ko select karte hain.
- Uske andar Permissions tab mein aapko ek option milta hai: "Add permissions" aur fir "Create inline policy".
- Wahan aap JSON code likhte hain aur save kar dete hain.

Yeh save hote hi usi user ke page par niche ek chote se drop-down box mein dikhne lagti hai. Yeh main IAM ke 'Policies' waale section mein kabhi nazaar nahi aayegi.

**Inline Policy Ke Fayde**:

Aap sochenge ki agar managed policy itni acchi hai, to AWS ne inline policy ka option diya hi kyun? Iske kuch specific fayde hain:
- Strict Permission Control (No Accidental Attachment): Maano aapke paas ek hile-dule ya bohot sensitive database ka access hai. Aap chahte hain ki yeh access sirf aur sirf "Senior-DBA" naam ke role ko mile. Agar aap managed policy banayenge, to ho sakta hai kal ko koi naya AWS admin galti se wo policy kisi normal developer se attach kar de. Lekin agar aapne inline policy banayi hai, to wo sirf usi Senior-DBA ke andar rahegi. Koi bhi dusra admin use galti se kisi aur user se attach nahi kar payega.
- No Policy Limits Overlap: AWS mein ek account mein maximum Customer Managed Policies banane ki ek limit hoti hai. Agar aap aisi permissions likh rahe hain jo sirf ek hi baar use honi hain, to inline policy banakar aap apni managed policy ki limit ko barbad hone se bacha sakte hain.

**Inline Policy Ke Nuksan**:

AWS hamesha kehta hai ki inline policy ka use kam se kam karein. Iske piche bohot bade reasons hain:
- No Reusability (Mehnat Zyada): Maano aapke office mein 10 developers hain aur sabko alag-alag permissions chahiye par unme se ek permission (jaise S3 Read) sabke liye common hai. Agar aap inline use karenge, to aapko 10 users ke andar ja kar 10 baar wo JSON code copy-paste karna padega. Isse aapka time barbad hoga.
- Management Nightmare (Centralization ka na hona): Maniye aapne 10 users ke andar inline policy daal di. Kal ko boss ne bola, "S3 Read ke sath-sath sabko S3 List ka access bhi de do." Ab aapko ek-ek karke pancho ya daso users ke permissions tab mein jana padega, unki inline policy ko edit karna padega aur badlav karna padega. Agar ek bhi user chhut gaya, to error aane lagega.
- No Version Control: Yeh iska sabse khatarnak point hai. Managed policy mein purane versions (v1, v2, v3) ka backup rehta hai. Inline policy mein koi version control nahi hota. Agar aapne inline policy ka JSON code edit kiya aur naya code save kar diya, to purana code hamesha ke liye gayab. Agar naye code mein koi galti hui, to aap pichle version par roll-back nahi kar sakte, aapko khud baith kar naya code sahi karna padega.

**Kab Kaunsi Policy Use Karein?**:
- Managed Policy Use Karein: Jab permissions common hon, aur aapko pata ho ki yeh aaj ek user ko chahiye to kal kisi aur ko bhi chahiye ho sakti hai (Jaise: Standard Developer Permissions, Read-Only access, etc.).
- Inline Policy Use Karein: Jab permission 100% unique ho, aur aap hamesha ke liye ensure karna chahte hain ki yeh permission kabhi bhi, galti se bhi kisi aur user ya role ko na di ja sake.

### Resource-Based Policy
