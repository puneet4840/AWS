# EC2 instance accessing S3 bucket using IAM Role

Hum ek real-world, production-level lab karenge. Is lab mein hum ek **Node.js/Python** Application (EC2 par) ko ek securely configure kiye gaye **AWS IAM Role** ke zariye ek Private S3 Bucket se connect karenge.

<br>

### Lab Architecture Overview

- Resource 1: Ek Private S3 Bucket (jahan secure files hain).
- Resource 2: Ek IAM Role (Trust + Permission policy ke saath).
- Resource 3: Ek EC2 Instance (jispar humara code chalega).

<br>
<br>

### Step 1: Private S3 Bucket Banana

Sabse pehle hum ek bucket banayenge jahan humari production files hongi.
- AWS Console mein **S3 Dashboard** par jayein aur **Create bucket** par click karein.
- Bucket Name dalein: ```prod-data-secure-bucket-2026``` (Naam unique hona chahiye).
- **Block Public Access settings** mein check karein ki **Block all public access** par tick ho (Production mein bucket hamesha private hoti hai).
- **Create bucket** par click kar dein.
- Is bucket ke andar ek dummy file upload kar dein, jaise: ```app-config.json```.

<br>
<br>

### Step 2: Production IAM Role Create Karna

Ab hum woh identity banayenge jise server adopt karega.
- **IAM Dashboard** mein jayein, left menu se **Roles** select karein aur **Create role** par click karein.
- **Trusted entity type** mein **AWS service** chunein aur Service or use case dropdown se **EC2** select karein. Next par click karein.
- **Permissions page** par hum AWS ki bani-banayi policy use nahi karenge (kyunki woh poore S3 ka access de deti hain). Hum apni custom policy banayenge. **Create policy** button par click karein (ek naya tab khulega).
- JSON tab select karein aur neeche di gayi production-ready policy wahan paste karein:
```
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "ProductionS3ReadAccess",
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::prod-data-secure-bucket-2026",
                "arn:aws:s3:::prod-data-secure-bucket-2026/*"
            ]
        }
    ]
}
```

Note: Yeh policy sirf isi ek specific bucket ko read karne ki taqat deti hai, baaki kisi bucket ko nahi.

- Policy ko naam dein ```Prod-S3-Read-Only-Policy``` aur **Create** kar dein.
- Purane Role wale browser tab par vapas aaein, refresh karein aur apni nayi bani policy ```Prod-S3-Read-Only-Policy``` ko select karke Next karein.
- Role ko naam dein: **Prod-Application-Execution-Role**.
- **Create Role** par click kar dein. (AWS background mein iski Trust policy automatic EC2 ke liye configure kar dega).

<br>
<br>

### Step 3: EC2 Instance Launch Karna (IMDSv2 Mandatory)

Ab hum apna server banayenge aur usme security strictly tight rakhenge.
- **EC2 Dashboard** par jayein aur **Launch instance** par click karein.
- Name dalein: ```Prod-Web-Server```.
- OS mein **Amazon Linux 2023** ya **Ubuntu** chunein.
- **Advanced Details** section ko expand karein (Yeh sabse important step hai):
  - **IAM instance profile**: Dropdown se apna banaya hua role **Prod-Application-Execution-Role** select karein.
  - **Metadata accessible**: Enabled chunein.
  - **Metadata version**: Production standard ke mutabik ise **V2 only** (Required) select karein. (Isse IMDSv1 disable ho jayega aur SSRF attacks ka khatra khatam ho jayega).
- **Launch Instance** par click kar dein.

<br>
<br>

### Step 4: Live Verification Aur Production Testing (The Real Magic)

Jab server chal jaye, toh usme SSH ya **EC2 Instance Connect** ke zariye terminal login karein. Ab hum wahan permanent keys ki jagah IAM Role ka live test karenge.

**Test A: Manual IMDSv2 Token Request (Peeche Ka Process Dekhna)**:

Kyunki humne IMDSv2 required kiya hai, hum direct URL hit nahi kar sakte. Pehle hume ek temporary session token banana hoga, fir metadata check karna hoga. Apne EC2 terminal par yeh do commands chalayein:
```
# Step A1: IMDSv2 ke liye 6 ghante ka ek secure token generate karein
TOKEN=$(curl -X PUT "http://169.254.169" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# Step A2: Us token ka use karke STS se aayi temporary keys ko live terminal par dekhein
curl -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169
```

**Output**: Jaise hi aap yeh run karenge, terminal par aapko ek JSON output dikhega jisme ```AccessKeyId``` (shuruwat ASIA se hogi), ```SecretAccessKey```, ```Token``` aur ```Expiration time``` saaf-saaf likha hoga. Yeh wahi credentials hain jo STS ne automatic generate ki hain.

<br>

**Test B: Application Level Test (Bina Kisi Key Ke Connection)**:

Ab terminal par AWS CLI ka use karke bucket se file read karne ki koshish karte hain. Humne server par koi login id ya password set nahi kiya hai.

```
# Bucket ke andar ki files ki list dekhne ke liye
aws s3 ls s3://prod-data-secure-bucket-2026
```

**Result**: Aapko ```app-config.json``` file dikh jayegi. AWS CLI ne background mein khud hi IMDSv2 se token uthaya, STS se verify kiya aur S3 se data lakar de diya.

<br>

**Test C: Security Boundary Test (Least Privilege Ka Proof)**:

Ab check karte hain ki kya yeh role apni limits se baahar ja sakta hai? Hum ek nayi file upload (Write) karne ki koshish karte hain, jabki humne permission sirf Read (```GetObject```) ki di thi.

```
# Ek dummy file banayein
echo "hack" > hack.txt

# Use upload karne ki koshish karein
aws s3 cp hack.txt s3://prod-data-secure-bucket-2026/
```

**Result**: Aapko terminal par upload failed: **An error occurred (AccessDenied)** ka error milega. Isse yeh proof hota hai ki production policy perfect kaam kar rahi hai—application sirf wahi kar sakti hai jiske liye use authorize kiya gaya hai.
